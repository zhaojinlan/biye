基于多智能体与知识图谱的智能医疗辅助诊断系统

创新点详细说明


一、技术创新

创新点1：基于LangGraph的多智能体协作架构

1.1 创新描述

本系统设计了一个专门面向医疗诊断场景的多智能体协作架构，包含5个专业化智能体节点，通过状态图管理实现智能协作。

核心组件：
- 监督节点（Supervisor）：任务分类与智能路由
- 分诊节点（Triage）：并行分诊评估
- 风险评估节点（Recommend）：疾病风险分析
- 多专家会诊节点（Experts）：多视角诊断
- 知识查询节点（Query）：医学知识问答

技术特点：
- 使用LangGraph的StateGraph实现状态管理
- 支持条件路由和动态任务分配
- 保持多轮对话上下文
- 支持并行处理和串行处理的混合模式

1.2 与主流方案对比

传统医疗AI系统：
- 单一大模型处理所有任务
- 缺乏专业化分工
- 难以处理复杂的多步骤诊疗流程
- 推理过程不透明

本系统优势：
- 多个专业化智能体，每个专注特定医疗任务
- 清晰的任务分工和协作机制
- 支持复杂的诊疗工作流
- 每个节点的决策过程可追溯

1.3 实现细节

状态定义：
- messages：对话历史
- patient_id：患者标识
- disease_data：疾病数据
- analysis_result：分析结果
- has_triaged：分诊状态标记

路由策略：
- 基于LLM的意图识别
- 上下文感知的路由决策
- 防止重复分诊的状态检查

工作流程：
1. 用户输入→监督节点
2. 监督节点分析任务类型
3. 路由到对应的专业节点
4. 专业节点处理并更新状态
5. 返回结果，支持继续对话

1.4 创新价值

- 提高诊断准确性：专业化分工减少错误
- 增强可解释性：每个节点的决策逻辑清晰
- 易于扩展：可灵活添加新的专业节点
- 符合医疗实践：模拟真实的医疗诊疗流程


创新点2：医学知识图谱自动化构建流程

2.1 创新描述

本系统实现了从PDF医学文献到知识图谱的全自动化构建流程，无需人工标注，大幅降低知识维护成本。

技术流程：
1. PDF智能解析：使用Docling保留文档结构
2. 文本智能清洗：去除无关内容（封面、参考文献等）
3. 文本智能分块：RecursiveCharacterTextSplitter
4. 实体自动抽取：基于LLM的零样本实体识别
5. 关系自动抽取：基于LLM的关系识别
6. 图谱自动存储：写入Neo4j并自动去重
7. 向量自动索引：创建症状向量索引

2.2 与主流方案对比

传统知识图谱构建方法：
- 人工标注：耗时耗力，成本高
- 规则匹配：需要大量规则，难以覆盖所有情况
- 监督学习：需要大量标注数据训练模型
- 更新困难：添加新知识需要重新标注和训练

本系统优势：
- 全自动化：从PDF到知识图谱一键完成
- 零样本学习：利用LLM的通用能力，无需标注数据
- 易于更新：上传新文献即可自动扩展知识图谱
- 保留结构：Docling保留文档结构信息，提高抽取准确率

2.3 实现细节

PDF解析（Docling）：
- 支持复杂排版的医学文献
- 识别标题、段落、表格、图片
- 输出结构化HTML
- 保留语义信息

文本清洗：
- 正则表达式识别DOI、参考文献
- BeautifulSoup解析HTML
- 去除空格、特殊字符
- 保留核心医学内容

实体抽取（LLM）：
- 提示词工程：设计专业的医学实体抽取提示词
- 结构化输出：使用JsonOutputParser确保输出格式
- 实体类型：疾病、症状、风险因子、病原体、治疗方法、检查方法
- 批量处理：对每个文本块进行实体抽取

关系抽取（LLM）：
- 识别实体间的医学关系
- 关系类型：HAS_SYMPTOM、HAS_RISK_FACTOR、HAS_TREATMENT等
- 上下文理解：基于文本块的语义理解

图谱存储（Neo4j）：
- 自动创建节点和关系
- 自动去重：相同实体合并
- 属性更新：补充实体描述信息
- 索引优化：创建必要的索引

向量索引（Neo4j Vector）：
- 症状文本向量化（M3E模型）
- 创建向量索引：enhanced_symptom_vectors
- 支持语义相似度检索

2.4 创新价值

- 降低成本：无需人工标注，节省大量人力
- 提高效率：自动化处理，快速构建知识图谱
- 易于维护：新增文献自动更新知识图谱
- 保证质量：基于LLM的抽取，准确率高


创新点3：基于MCP协议的微服务架构

3.1 创新描述

本系统采用新兴的模型上下文协议（MCP）实现服务间通信，实现了轻量级客户端和重量级服务端的分离。

架构设计：
- 客户端：仅包含智能体逻辑，轻量级（约50MB依赖）
- 服务端：包含所有重量级依赖（约2GB依赖）
- 通信：通过MCP协议调用服务端工具
- 部署：客户端可本地部署，服务端云端部署

MCP工具：
- symptom_search_analyze：症状检索与分析
- query_disease_info：疾病信息查询
- get_diagnostic_tests：诊断检查方法查询
- query_treatment_methods：治疗方法查询
- rag_search：文档向量检索

3.2 与主流方案对比

传统微服务架构：
- RESTful API：需要手动定义每个接口
- gRPC：需要定义protobuf文件
- GraphQL：需要定义schema
- 客户端需要了解所有接口细节

本系统优势：
- 标准化协议：MCP提供统一的工具调用接口
- 工具发现：客户端自动发现可用工具
- 动态加载：服务端可动态添加新工具
- 轻量级客户端：客户端无需包含重量级依赖

3.3 实现细节

MCP服务端（FastMCP）：
- 使用@mcp.tool()装饰器定义工具
- 自动生成工具描述和参数schema
- 支持SSE传输协议
- 提供工具列表和调用接口

MCP客户端（langchain-mcp-adapters）：
- MultiServerMCPClient连接多个MCP服务
- 自动获取工具列表
- 将MCP工具转换为LangChain工具
- 集成到智能体工作流

通信流程：
1. 客户端连接MCP服务端（SSE）
2. 客户端请求工具列表
3. 服务端返回工具定义
4. 智能体调用工具
5. 客户端通过MCP协议发送请求
6. 服务端执行工具逻辑
7. 返回结果给客户端

部署优势：
- 客户端轻量级，可打包为可执行文件
- 服务端集中部署，统一管理
- 降低客户端硬件要求（无需GPU）
- 易于更新和维护

3.4 创新价值

- 降低部署成本：客户端无需高配置硬件
- 提高可维护性：服务端集中管理，易于更新
- 增强可扩展性：易于添加新的工具服务
- 标准化接口：遵循MCP协议标准


创新点4：混合检索增强生成系统

4.1 创新描述

本系统结合Neo4j知识图谱和Redis向量数据库，实现了结构化知识和非结构化文档的混合检索。

检索策略：
- 结构化检索：基于Neo4j知识图谱的精确查询
- 向量检索：基于Redis的语义相似度检索
- 混合检索：根据查询类型自动选择最优策略
- 多跳推理：症状→疾病→风险因子→治疗方法

技术实现：
- Neo4j：存储医学知识图谱，支持Cypher查询
- Redis：存储文档向量，支持相似度检索
- M3E模型：中文医学文本嵌入
- 检索融合：结合两种检索结果

4.2 与主流方案对比

单一RAG系统：
- 仅使用向量检索
- 缺乏结构化推理能力
- 难以处理复杂的关系查询
- 检索结果可能不准确

纯知识图谱系统：
- 仅支持精确查询
- 难以处理模糊查询
- 无法检索非结构化文档
- 知识覆盖有限

本系统优势：
- 双引擎检索：既能精确查询又能语义匹配
- 结构化推理：支持多跳推理和关系查询
- 知识互补：结构化知识+非结构化文档
- 智能选择：根据查询类型自动选择最优策略

4.3 实现细节

知识图谱检索（Neo4j）：
- 症状查询疾病：
  MATCH (d:Disease)-[:HAS_SYMPTOM]->(s:Symptom)
  WHERE s.name CONTAINS $symptom
  RETURN d
- 疾病查询风险因子：
  MATCH (d:Disease)-[:HAS_RISK_FACTOR]->(rf:RiskFactor)
  WHERE d.name = $disease
  RETURN rf
- 多跳查询：
  MATCH path = (s:Symptom)<-[:HAS_SYMPTOM]-(d:Disease)-[:HAS_TREATMENT]->(t:Treatment)
  RETURN path

向量检索（Redis）：
- 文档向量化：使用M3E模型
- 向量索引：RediSearch FLAT索引
- 相似度计算：余弦相似度
- Top-K检索：返回最相似的K个结果

混合检索策略：
- 精确查询：使用知识图谱（如查询特定疾病的治疗方法）
- 模糊查询：使用向量检索（如症状描述匹配）
- 复杂查询：先知识图谱后向量检索（如症状→疾病→文档）
- 结果融合：合并两种检索结果，去重和排序

应用场景：
- 分诊节点：症状向量检索+知识图谱查询
- 风险评估节点：知识图谱多跳推理
- 多专家会诊节点：向量检索相关文献
- 知识查询节点：混合检索回答问题

4.4 创新价值

- 提高检索准确率：结合精确查询和语义匹配
- 增强推理能力：支持复杂的多跳推理
- 扩大知识覆盖：结构化知识+非结构化文档
- 灵活适应：根据查询类型自动选择最优策略


二、应用创新

创新点5：并行分诊机制

5.1 创新描述

本系统设计了两个并行工作的分诊智能体，分别从医学顾问和急诊分诊两个角度评估患者，提供更全面的分诊建议。

智能体1（医学顾问）：
- 基于MCP工具进行症状分析
- 查询知识图谱获取相关疾病
- 评估风险因子
- 提供医学建议

智能体2（急诊分诊）：
- 基于五级分诊标准
- 识别危重指征
- 评估紧急程度
- 推荐就诊科室

综合分析：
- 整合两个智能体的结果
- 生成综合分诊报告
- 提供补充问题建议

5.2 创新价值

- 多视角评估：医学和急诊两个角度
- 提高准确性：互相验证，减少误判
- 符合规范：遵循五级分诊标准
- 实用性强：直接指导急诊工作


创新点6：动态风险评估

6.1 创新描述

本系统基于知识图谱实时计算患者症状与疾病风险因子的匹配度，使用Softmax算法量化疾病概率。

评估流程：
1. 症状向量检索：找到相似症状
2. 疾病查询：获取相关疾病列表
3. 风险因子匹配：计算匹配的风险因子数量
4. 概率计算：使用Softmax算法
5. 检查建议：基于疾病推荐检查

算法实现：
- 得分计算：score = matched_risk_factors / total_risk_factors
- Softmax概率：P(disease_i) = exp(score_i) / Σexp(score_j)
- 置信度：最高概率疾病的概率值

6.2 创新价值

- 量化评估：提供可解释的概率值
- 动态计算：基于患者实际情况
- 支持多疾病：同时评估多个候选疾病
- 指导检查：推荐针对性的诊断检查


创新点7：多专家会诊模拟

7.1 创新描述

本系统设计了三个专业化智能体模拟多学科专家会诊，每个专家基于RAG检索的专业知识独立给出意见。

专家角色：
- 诊断专家：分析症状和检查结果，给出诊断意见
- 治疗专家：基于诊断结果提供治疗方案
- 影像专家：解读影像学检查结果

工作流程：
1. RAG检索：为每个专家检索相关知识
2. 并行分析：三个专家独立分析
3. 综合报告：整合多专家意见
4. 一致性检查：识别意见分歧

7.2 创新价值

- 多维度分析：诊断、治疗、影像多角度
- 弥补盲区：单一医生的知识盲区
- 提高质量：模拟真实的多学科会诊
- 适用基层：为基层医疗提供专家级支持
