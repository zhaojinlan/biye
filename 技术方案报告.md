## 摘要

急诊医疗场景长期面临患者多、病情复杂、时间紧迫的挑战，医生难以在有限时间内全面获取最新诊疗知识，跨学科会诊资源更是严重不足。本方案针对急诊辅助诊断这一工程化医疗难题，提出了一种**基于多智能体协作与知识图谱融合的智能医疗辅助诊断系统**。

系统核心思想是：以大语言模型（Qwen2.5-14B）为推理引擎，通过 LangGraph 框架构建包含监督节点、分诊节点、风险评估节点、多专家会诊节点和知识查询节点的**五节点多智能体协作架构**，模拟真实医疗诊疗流程中的分诊—评估—会诊—查询全链路。在知识支撑层面，设计了从 PDF 医学文献到 Neo4j 知识图谱的**七步全自动化构建流水线**（PDF解析→文本清洗→Markdown转换→智能分块→LLM实体关系抽取→图谱存储→向量化索引），实现零人工标注的知识积累。同时，创新性地采用**混合检索增强生成（Hybrid RAG）** 策略，融合 Neo4j 结构化知识图谱精确查询与 Redis 向量语义检索，并通过新兴的**模型上下文协议（MCP）** 实现服务间标准化工具调用与轻重端分离部署。

预期方案能够为急诊医生提供秒级智能分诊、量化疾病风险评估、多专家视角会诊分析和实时医学知识检索，有效降低基层医疗机构对专家资源的依赖，提升急诊诊疗效率与准确性。系统已完成全功能原型开发，覆盖4个急诊常见疾病领域的知识图谱，具备良好的可扩展性和实际落地潜力。

## 1. 问题定义与总体思路

### 1.1 对工程问题的理解与定义

#### 1.1.1 问题背景

2025年11月，国家卫生健康委等五部门联合发布《关于促进和规范"人工智能+医疗卫生"应用发展的实施意见》，明确提出到2027年形成一批临床专病专科垂直大模型和智能体应用，基层诊疗智能辅助在医疗卫生机构广泛应用；到2030年，基层诊疗智能辅助应用基本实现全覆盖。政策为AI赋能医疗提供了明确的方向和强力支撑。

然而，急诊医疗作为医疗体系中最具时间敏感性和复杂性的场景，长期面临以下工程化挑战：

#### 1.1.2 核心痛点

**（1）急诊分诊效率瓶颈**

急诊科日均接诊量大、病种覆盖广，传统分诊依赖护士个人经验，面对多学科交叉的复杂病情容易出现误判或延误。五级分诊标准的执行缺乏统一的智能辅助工具，分诊质量参差不齐。

**（2）医学知识获取困境**

医学知识更新迅速，诊疗指南、专家共识等权威文献数量庞大且分散。医生在紧急诊疗过程中难以快速检索到所需的精准知识，导致诊疗决策可能滞后于最新医学进展。

**（3）跨学科会诊资源稀缺**

复杂病例需要多学科专家协同分析，但专家资源有限、协调困难，尤其在基层医疗机构，专家资源严重不足，患者难以获得多维度的诊断评估。

**（4）医学知识结构化管理缺失**

医院积累的病例经验和文献知识难以系统化、结构化管理，传统人工整理成本高昂，知识复用率低，大量宝贵的临床经验随着人员流动而流失。

#### 1.1.3 问题定义与定位

本方案聚焦于以下核心工程问题：**如何构建一个融合多智能体协作与知识图谱的智能系统，实现急诊场景下从医学文献自动化知识构建到智能分诊、风险评估、多专家会诊的全链路辅助诊断？**

问题边界：以急诊常见疾病（坏死性软组织感染、急性心力衰竭、急性心房颤动等）为知识域，面向急诊科医生和基层全科医生，提供辅助诊断建议而非替代医生决策。

核心挑战：
- 如何实现从非结构化PDF医学文献到结构化知识图谱的全自动化转换？
- 如何设计多智能体协作架构以模拟真实诊疗流程中的分工与协同？
- 如何融合结构化知识图谱检索与向量语义检索以抑制大模型"幻觉"？
- 如何实现轻量级客户端与重量级服务端的解耦部署以适配基层硬件条件？

#### 1.1.4 AI介入的必要性

上述问题的本质是**海量非结构化医学知识的理解、组织与智能应用**——这恰恰是大语言模型、知识图谱和多智能体系统的核心能力交汇点。传统规则引擎和专家系统难以处理自然语言文献、难以适应知识更新、难以模拟多角色协作，而AI技术的突破使得这些问题首次具备了工程化解决的可行性。

## 2. 详细AI技术方案

### 2.1 AI模型/算法设计

#### 2.1.1 多智能体协作架构（核心创新）

本系统基于LangGraph框架设计了**五节点多智能体协作架构**，通过有向状态图实现智能体间的数据流转和任务协调。

```
                    ┌──────────┐
                    │  START   │
                    └────┬─────┘
                         ▼
                ┌────────────────┐
                │   监督节点      │  ← LLM意图识别
                │ supervisor_node│     任务分类与路由
                └───┬──┬──┬──┬──┘
       症状描述 │  │  │  │ 无需处理
          ┌─────┘  │  │  └─────┐
          ▼        │  │        ▼
   ┌────────────┐  │  │   ┌────────┐
   │  分诊节点   │  │  │   │  END   │
   │ triage_node│  │  │   └────────┘
   └────────────┘  │  │
          病史/个人情况  检查结果  知识查询
               │  │
               ▼  ▼
   ┌────────────┐  ┌────────────┐  ┌────────────┐
   │ 风险评估节点│  │多专家会诊  │  │ 知识查询节点│
   │recommend   │  │agen_node   │  │ other_node │
   └────────────┘  └────────────┘  └────────────┘
```

**各节点设计：**

| 节点 | 职责 | 核心技术 | 输出 |
|------|------|----------|------|
| 监督节点 | 任务分类与智能路由 | LLM意图识别 + 上下文感知路由 | 路由决策 |
| 分诊节点 | 并行分诊评估 | 双智能体并行（医学顾问+急诊分诊） | 五级分诊报告 |
| 风险评估节点 | 疾病风险量化分析 | 知识图谱匹配 + Softmax概率计算 | 疾病概率分布+检查建议 |
| 多专家会诊节点 | 多维度诊断分析 | 三专家并行（诊断/治疗/影像）+ RAG | 综合会诊报告 |
| 知识查询节点 | 医学知识问答 | ReAct智能体 + MCP工具调用 | 知识查询结果 |

**状态管理设计：**

系统使用 TypedDict 定义共享状态，包含 `messages`（对话历史）、`patient_id`（患者标识）、`disease_data`（疾病匹配数据）、`has_triaged`（分诊标记）、`has_diagnosis`（诊断标记）等字段，实现多轮对话上下文的持续传递。

**智能路由机制：**

监督节点不仅基于LLM进行意图识别，还实现了**上下文感知路由**——若患者已完成分诊但尚未进行诊断，系统会自动强制路由至风险评估节点，模拟真实诊疗流程中的阶段递进。

#### 2.1.2 并行分诊算法

分诊节点采用创新的**双智能体并行评估**机制：

- **智能体A（医学顾问）**：通过MCP工具调用知识图谱进行症状分析，从疾病谱层面评估患者情况
- **智能体B（急诊分诊）**：基于中国急诊五级分诊标准（Ⅰ级-濒危、Ⅱ级-危重、Ⅲ级-急症、Ⅳ级-亚急症、Ⅴ级-非急症），评估患者紧急程度

两个智能体并行执行后，由**综合分析节点**融合两方意见，生成包含分诊级别、建议科室、危重指征、补充问询建议的完整分诊报告。

#### 2.1.3 Softmax风险评估算法

风险评估节点实现了基于知识图谱的**量化疾病风险评估**：

1. **症状提取**：LLM从患者描述中提取症状关键词
2. **症状-疾病匹配**：通过Neo4j向量相似度检索，查找与患者症状关联的候选疾病
3. **风险因子评分**：对每个候选疾病，计算患者匹配的风险因子数量占该疾病总风险因子的比例

   `score_i = matched_risk_factors_i / total_risk_factors_i`

4. **Softmax概率归一化**：

   `P(disease_i) = exp(score_i) / Σ_j exp(score_j)`

5. **检查推荐**：基于Top-N最可能疾病，通过MCP工具查询对应的诊断检查项目

#### 2.1.4 多专家会诊机制

多专家会诊节点模拟真实多学科会诊（MDT），设计三个专业化智能体：

- **诊断专家**：分析症状和检查结果，给出诊断意见
- **治疗专家**：基于诊断结果提供治疗方案
- **影像专家**：解读影像学检查结果

每个专家在给出意见前，先通过RAG从医学文献中检索相关知识段落作为参考依据，确保意见有据可循。三位专家独立分析后，由综合节点整合多方意见，生成结构化的会诊报告。

#### 2.1.5 知识图谱自动化构建算法

设计了**七步全自动化流水线**，实现从PDF文献到知识图谱的零人工标注转换：

```
PDF文献 → [1.Docling解析] → HTML → [2.BeautifulSoup清洗] → 清洁HTML
→ [3.html2text转换] → Markdown → [4.RecursiveCharacterTextSplitter分块]
→ 文本块 → [5.LLM实体关系抽取] → 结构化JSON
→ [6.py2neo写入Neo4j] → 知识图谱 → [7.M3E-Base向量化] → 向量索引
```

**实体关系抽取**采用LLM零样本学习，通过精心设计的结构化Prompt指导模型从每个文本块中抽取6类实体（Disease、Symptom、RiskFactor、Pathogen、Treatment、DiagnosticTest）和5类关系（HAS_SYMPTOM、HAS_RISK_FACTOR、CAUSED_BY、TREATED_BY、DIAGNOSED_BY），输出为JSON格式，使用Pydantic模型进行校验。

### 2.2 数据处理策略

#### 2.2.1 数据来源

| 数据类别 | 来源 | 类型 | 说明 |
|----------|------|------|------|
| 医学文献 | 中华急诊医学杂志等权威期刊 | PDF格式的专家共识/诊疗指南 | 如《坏死性软组织感染临床诊治急诊专家共识》《急性心力衰竭中国急诊管理指南（2022）》《急性心房颤动中国急诊管理指南（2024）》等 |
| 医学本体 | UMLS、ICD-11编码体系 | 标准化医学术语 | 用于实体归一化参考 |
| 测试数据 | 团队根据临床教材构造 | 模拟病例 | 用于系统功能验证 |

#### 2.2.2 数据预处理流程

1. **PDF解析**：使用IBM开源的Docling工具，保留文档结构信息（标题层级、段落、表格），输出结构化HTML
2. **文本清洗**：
   - 正则表达式识别并去除DOI、封面信息、参考文献列表
   - BeautifulSoup解析HTML，去除噪声标签和冗余空格
   - 保留核心医学内容（疾病描述、症状特征、治疗方案等）
3. **文本分块**：使用LangChain的RecursiveCharacterTextSplitter，参数设置为 `chunk_size=2000, chunk_overlap=200`，按标题和段落边界智能切分，确保每个块的语义完整性
4. **向量化**：使用M3E-Base模型（针对中文医学文本优化的768维嵌入模型）将文本块转化为向量表示

#### 2.2.3 数据伦理考量

- 本系统使用的医学文献均为**公开发表的学术论文和诊疗指南**，不涉及患者隐私数据
- 系统定位为**辅助诊断工具**，明确标注"仅供参考，不替代医生决策"
- 测试数据为模拟构造，不包含真实患者信息
- 患者数据采用本地存储、隔离管理，不上传至云端

### 2.3 系统架构与关键技术细节

#### 2.3.1 微服务架构设计

系统采用**客户端-服务端-数据层-模型层**四层微服务架构：

| 层级 | 组件 | 技术栈 | 部署方式 |
|------|------|--------|----------|
| 客户端层 | 多智能体引擎 + Vue3诊断界面 | Python、LangChain 0.3.27、LangGraph 0.6.8、FastAPI、Vue3 | 本地运行（Electron打包） |
| 服务端层 | REST API + MCP工具 + 知识构建 + Vue3管理界面 | Python、FastAPI 0.117.1、FastMCP、Docling 2.0、Vue3、ECharts | 云服务器（Docker） |
| 数据层 | 知识图谱 + 向量索引 + 文件存储 | Neo4j 5.14+、Redis 5.0+（RediSearch）、文件系统 | Docker容器 |
| 模型层 | LLM推理服务 | Ollama、Qwen2.5-14B | AutoDL云GPU + cpolar穿透 |

#### 2.3.2 MCP协议创新应用（关键技术亮点）

本系统采用Anthropic发布的**模型上下文协议（MCP）** 实现服务间通信，这是一项前沿的AI基础设施协议。

**MCP在本系统中的角色：**

```
客户端智能体(LangGraph) → MCP Client(langchain-mcp-adapters)
        ↕ SSE长连接
服务端(FastMCP) → 4个标准化医学知识工具 → Neo4j/Redis
```

**4个MCP工具定义：**

| 工具名 | 功能 | 数据源 |
|--------|------|--------|
| `symptom_search_analyze` | 症状向量检索与疾病关联分析 | Neo4j向量索引 |
| `get_common_diagnostic_methods` | 查询通用诊断检查方法 | Neo4j图谱 |
| `get_diagnostic_tests_for_disease` | 查询特定疾病的诊断检查 | Neo4j图谱 |
| `retrieve_medical_knowledge_vector` | 向量语义检索医学文献 | Redis向量索引 |

**MCP的工程价值：** 客户端无需包含Neo4j、Redis等重量级依赖（约50MB vs 2GB），通过标准化协议动态发现和调用服务端工具，实现了真正的轻重端分离，使客户端可打包为桌面应用部署于基层医疗机构的普通电脑上。

#### 2.3.3 混合检索增强生成（Hybrid RAG）

本系统融合两种互补的检索策略：

| 检索通道 | 技术 | 适用场景 | 优势 |
|----------|------|----------|------|
| 知识图谱检索 | Neo4j Cypher查询 | 精确查询（如：某疾病有哪些症状） | 结构化推理、多跳查询 |
| 向量语义检索 | Redis + M3E-Base 768维向量 | 模糊匹配（如：根据症状描述找相关文献） | 语义理解、覆盖面广 |

不同智能体节点根据任务特点自动选择最优检索策略：
- **风险评估**：主要使用知识图谱（精确的症状-疾病-风险因子多跳推理）
- **多专家会诊**：主要使用向量检索（获取相关文献支撑专家意见）
- **知识查询**：混合使用（先图谱精确查询，再向量补充检索）

#### 2.3.5 Neo4j知识图谱数据模型

```
    ┌──────────┐                    ┌──────────┐
    │ Symptom  │◄─ HAS_SYMPTOM ───│ Disease  │
    │ 症状     │                    │ 疾病     │
    └──────────┘                    └────┬─────┘
                                    │  │  │  │
                          CAUSED_BY │  │  │  │ HAS_RISK_FACTOR
                                    ▼  │  │  ▼
                              ┌────────┐│┌──────────┐
                              │Pathogen│││RiskFactor│
                              │病原体  │││风险因子   │
                              └────────┘│└──────────┘
                              TREATED_BY│  DIAGNOSED_BY
                                        ▼  ▼
                              ┌──────────┐┌──────────────┐
                              │Treatment ││DiagnosticTest│
                              │治疗方法  ││诊断检查       │
                              └──────────┘└──────────────┘
```

- **6种节点类型**：Disease、Symptom、RiskFactor、Pathogen、Treatment、DiagnosticTest
- **5种关系类型**：HAS_SYMPTOM、HAS_RISK_FACTOR、CAUSED_BY、TREATED_BY、DIAGNOSED_BY
- **向量索引**：enhanced_symptom_vectors（768维，余弦相似度）

## 3. 方案创新性与可行性分析

### 3.1 方案核心创新点

#### 创新点一：基于LangGraph的医疗多智能体协作架构

**创新之处：** 现有医疗AI系统多为单一大模型处理所有任务，缺乏专业化分工。本系统首次将LangGraph状态图框架应用于急诊诊疗场景，设计了五节点协作架构，每个节点专注特定医疗任务，通过智能路由和共享状态实现协同工作。

**颠覆性思路：**
- **监督者-工作者**协作模式模拟了真实医院的**导诊台→各科室**工作流
- **并行双智能体分诊**突破了传统单一评估的局限
- **上下文感知路由**实现了诊疗流程的阶段递进管理
- 每个节点的决策过程可追溯、可解释

**与主流方案对比：**

| 维度 | 传统医疗AI | 本系统 |
|------|-----------|--------|
| 架构 | 单一大模型端到端 | 五节点专业化分工 |
| 分诊 | 单一角色评估 | 双智能体并行+融合 |
| 推理透明度 | 黑盒 | 每个节点可追溯 |
| 可扩展性 | 需重新训练 | 添加新节点即可 |

#### 创新点二：知识图谱全自动化构建流水线

**创新之处：** 传统知识图谱构建依赖大量人工标注或需要预训练的NER/RE模型。本系统利用LLM的零样本学习能力，设计了七步全自动化流水线，从PDF文献到知识图谱一键完成，实现零标注数据、零额外训练的知识构建。

**技术突破：**
- Docling保留文档结构信息，提高下游抽取准确率
- LLM结构化Prompt + Pydantic校验，保障输出质量
- MERGE策略自动去重合并，支持增量更新
- 异步处理 + SSE实时进度推送，用户体验友好

#### 创新点三：混合检索增强生成 + MCP标准化工具调用

**创新之处：** 将结构化知识图谱检索与向量语义检索融合为统一的RAG体系，并通过MCP协议实现标准化封装。不同于单一RAG系统只能做语义匹配，本系统能同时支持精确的图谱多跳推理和模糊的语义关联检索。

**跨领域技术融合：**
- 知识图谱（精确推理） + 向量检索（语义匹配） = 互补增强
- MCP协议（AI基础设施） + 医疗知识服务 = 标准化工具生态
- 根据查询类型智能切换检索策略，最大化检索效果
